Organizational Success Factors

  1. Your IT organization widely recognizes IT as a competitive advantage to your business
  2. Your organization's leadership team recognizes IT as a competitive advantage
  3. Clearly defined business champions lead change
  4. Clearly defined technical champions lead change
  5. A project with measurable business value been selected to be automatically deployed to production
  6. There is a clear understanding of why the business is undertaking a technology transformation

Organizational Culture

  1. Information is actively sought in your organization
  2. Responsibilities and risks are shared
  3. Cross-functional collaboration is encouraged and rewarded
  4. New ideas and innovations are welcomed
  5. There a strong internal community
  6. Blameless postmortems/retrospectives are required for success and failure
  7. Lessons learned are posted in a highly visible way (internal blog, email DL, etc)
  8. Different teams, such as Development, Operations, Testing, Security, and Project Management, have shared goals and methodologies

Data-Driven Decision Making

  1. Your organization has quantifiable goals
  2. Your organization's goals are visible to all members of the organization
  3. Data is collected and used to make decisions
  4. You have cost and utilization monitoring in place
  5. Monitoring data from pre-production environments is used to make release decisions
  6. Monitoring provides business performance information

Coding Practices

  1. Code reviews are performed and results are shared with developers
  2. Regularly scheduled automation demos occur
  3. Any potential contributor to a project can find the code and documentation with minimal assistance
  4. Every change is reviewed by at least two people with relevant skill and contextual knowledge
  5. Developers follow code inspection standards (rubocop, foodcritic, etc.)

Version Control

  1. All source code is stored in a version control system (VCS)
  2. All infrastructure and deployment automation code stored in a VCS
  3. Software and automation components are shared and co-developed
  4. Developers can view projects in VCS with ease
  5. Developers suggest and/or provide changes to projects in VCS
  6. Developers develop and ship code for projects in VCS with few hurdles
  7. New projects are created in VCS with ease
  8. VCS makes code/project ownership and documentation clear
  9. Code documentation is easy to write and is viewable by all with VCS access

Chef Local Development

  1. Developers provision their own isolated VMs as needed
  2. Developers install Chef development tools (ChefDK, vagrant, etc)
  3. Chef Development workstation setup is automated
  4. Developers download code dependencies in a friction-free manner
  5. Developers use Vagrant boxes (or VM Images) which closely resemble production systems
  6. Developers run unit tests (ChefSpec) locally
  7. Developers run functional tests (ServerSpec) locally
  8. Developers run Test Kitchen to verify that cookbooks provide the desired functionality

Continuous Integration

  1. All projects use a CI service
  2. The CI service can test multiple branches of the same project
  3. The CI service automatically tests all new branches
  4. CI job templates have been defined for each type of software project (Chef cookbook, Java app servers, Node.js app, etc.)
  5. CI job templates include a code analysis stage (linting, code coverage, complexity, etc.)
  6. CI job templates include a unit testing stage
  7. CI job templates include an integration testing stage
  8. CI job templates include a functional testing stage
  9. Your CI tool makes the quality of your code base highly visible
  10. CI enforces code quality standards for each type of software project
  11. CI confirms that versions are unique
  12. CI tools automatically update cookbook dependencies at the start of a run
  13. Monitoring is integrated with CI, providing awareness of each deployment's effect on system health

Chef Code Deployment

  1. CI jobs upload cookbooks to a Chef server
  2. Cookbook updates are only be uploaded via CI
  3. All non-cookbook Chef policy (environments, roles, data bags, etc.) are only uploaded via CI
  4. CI jobs pin dependencies so that they cannot be modified in later deployments
  5. CI jobs assign a set of cookbook versions to a Chef environment
  6. Cookbook deployments are automated
  7. Cookbook deployment automation manages the sequence of deployments

Application Deployment

  1. Applications are deployed without manual intervention
  2. Applications follow a clear promotion path (e.g., Dev -> QA -> Staging -> Production)
  3. Application deployment automation manages the sequence of deployments (e.g.: Database schema first, then app servers)
  4. Application deployment automation performs parallel, rolling, and/or canary deployments
  5. Deployments are run during the business day without causing negative user experiences

Continuous Delivery

  1. All validation and deployments are executed in a pipeline that goes from source control all the way to production
  2. Small batches of work flow through the pipeline
  3. Changes are released weekly, if not more frequently
  4. Changes that pass validation are automatically released to production

Virtualizaton as a Service

  1. All servers are provisioned via APIs
  2. The API is a generally accepted API such as EC2, Azure, OpenStack, vSphere, Docker
  3. Access restrictions allow authorized users (e.g., developers and operations) to provision resources, and deny unauthorized users
  4. Resources are provisioned in a friction-free manner
  5. System images are built via an automated process
  6. System images are built from scratch or from a well known, trusted origin
  7. System images are built in a pipeline
  8. System images are freqently updated

Full-Stack Automation

  1. Storage resources are provisioned by code
  2. Networking resources are provisioned by code
  3. Identity services are managed by code
  4. DNS is configured by code
  5. Messaging queues are provisioned by code
  6. Security validation is performed by code
  7. Performance validation is performed by code

Sustaining Operations Culture

  1. The codebase is almost always in a releasable state
  2. Compliance is checked automatically
  3. Compliance automation removes non-compliant nodes from production upon detection
  4. Nodes are destroyed beyond a certain age (e.g. 30 days)
  5. Automated services stress the production system (e.g. Chaos Monkey)
  6. The entire application is test provisioned and deployed in an alternate datacenter
  7. Prodution alerts first notify those who wrote the code
  8. Operations, Security, Network, Automation, Testing, and Compliance experts provide tools and coaching to enable development teams
  9. Operations, Security, Network, Automation, Testing, and Compliance experts are available to support development teams in issue resolution
